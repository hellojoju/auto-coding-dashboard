# AI全自动开发平台 - Dashboard 实施计划 V2

> 基于 `Dashboard 设计稿 V2`
> 日期: 2026-04-18
> 状态: 建议稿

---

## 1. 计划目标

V2 计划的核心目标是先解决系统一致性问题，再做 UI 完整度。实施顺序必须围绕以下优先级：

1. 先统一状态源
2. 再打通命令闭环
3. 再稳定事件与恢复链路
4. 最后补强 UI 和多实例能力

---

## 2. 里程碑总览

1. Milestone 0: 状态模型收敛
2. Milestone 1: Repository 与快照落地
3. Milestone 2: 命令通路落地
4. Milestone 3: 事件流与 WebSocket 落地
5. Milestone 4: UI 接真实状态
6. Milestone 5: 多实例隔离模式
7. Milestone 6: 验收与回归

---

## 3. Milestone 0 - 状态模型收敛

### 目标

明确 Dashboard 使用的统一数据模型，避免继续在不同模块中各写一套结构。

### 任务

1. 定义 `feature`、`agent`、`command`、`event`、`snapshot` schema
2. 统一 `project_id`、`run_id`、`schema_version` 字段约定
3. 统一 `features.json` 的结构，不再允许读取和写出不同格式
4. 约束 Dashboard 不直接拥有独立业务状态源

### 验收

1. 所有设计文档都引用同一套字段命名
2. 所有状态相关文件都能明确归属到 project scope
3. 前后端契约文档不再出现冲突字段

---

## 4. Milestone 1 - Repository 与快照落地

### 目标

建立统一的持久化入口，让 PM、Dashboard、事件推送都围绕同一状态源工作。

### 任务

1. 引入 `ProjectStateRepository`
2. 将 features、agents、chat、events、commands 的读写统一收口
3. 提供快照生成逻辑
4. 提供 `last_event_id` 游标
5. 约束所有业务状态更新先落库，再广播

### 验收

1. 快照可单独生成
2. 页面刷新后可仅靠快照恢复基本状态
3. EventBus 不再承担真源职责

### 测试重点

1. repository 读写一致性
2. snapshot 生成正确性
3. schema 兼容性

---

## 5. Milestone 2 - 命令通路落地

### 目标

把 Dashboard 按钮从“发通知”升级为“创建并追踪命令”。

### 任务

1. 设计命令创建接口
2. 为命令分配 `command_id`
3. 为命令实现状态机
4. 由 PM 或 command processor 消费命令
5. 为命令执行结果产出事实事件

### 首批支持命令

1. `approve_decision`
2. `reject_decision`
3. `send_pm_message`
4. `pause_agent`
5. `resume_agent`
6. `retry_feature`
7. `skip_feature`

### 验收

1. 任意命令都能查询最终状态
2. 命令失败时能返回失败原因
3. 前端可以看到命令从 `pending` 到 `applied` 或 `failed`

### 测试重点

1. 命令状态迁移
2. 重复命令处理
3. 无效目标处理
4. 命令审计链路

---

## 6. Milestone 3 - 事件流与 WebSocket 落地

### 目标

建立可恢复、可补偿的实时推送链路。

### 任务

1. 为事件引入 `event_id`
2. 所有事件统一带 `project_id` 和 `run_id`
3. WebSocket 连接成功后支持按 `last_event_id` 补增量
4. 断线后允许重新拉快照并校正
5. 调整 EventBus，使其仅广播已提交事件

### 验收

1. 前端断开重连后不会遗漏关键事件
2. 如果事件游标不连续，系统能触发全量同步
3. 服务重启后仍可通过快照和事件恢复状态

### 测试重点

1. 事件顺序
2. 事件补发
3. WebSocket 重连
4. 快照校正

---

## 7. Milestone 4 - UI 接真实状态

### 目标

移除主页面对 mock 数据的依赖，让前端基于真实快照和事件工作。

### 任务

1. 页面初始化先请求快照
2. 再建立 WebSocket 连接
3. 本地 store 改成“快照 + 增量事件”模式
4. 所有控制按钮改成调用命令接口
5. 展示命令执行状态

### 验收

1. 主页面不依赖默认 mock 数据
2. 刷新后状态与后端一致
3. 按钮点击后不会直接伪造业务结果

### 测试重点

1. 首屏加载
2. 事件驱动更新
3. 命令回执展示
4. 页面刷新恢复

---

## 8. Milestone 5 - 多实例隔离模式

### 目标

只在主链路稳定后再引入多实例，并且默认使用隔离工作区。

### 任务

1. 为实例增加 workspace 标识
2. 定义单实例与多实例两种运行模式
3. 多实例下使用隔离工作区
4. 逻辑锁只保留为辅助调度信号
5. 合并前统一做集成验证

### 验收

1. 多实例不会共享写入同一工作区
2. 每个实例的产出可单独追溯
3. 集成失败能定位到实例来源

### 测试重点

1. workspace 分配
2. feature 与实例归属
3. 合并前验证
4. 回滚与失败隔离

---

## 9. Milestone 6 - 验收与回归

### 目标

用真实工作流验证 Dashboard 已从演示界面升级为控制台。

### 场景

1. 单实例开发流程
2. 用户中途审批
3. 用户中途暂停和恢复
4. 失败后重试
5. 页面刷新恢复
6. WebSocket 断线重连

### 验收标准

1. UI 状态与 PM 实际执行状态一致
2. 每个用户操作都有命令记录和结果
3. 服务重启后仍能恢复主要状态
4. 多实例能力默认关闭时，不影响单实例稳定性

---

## 10. 不建议的实施顺序

以下做法不建议继续：

1. 先做完整 UI 再补状态真源
2. 先上多实例再解决隔离问题
3. 先堆 WebSocket 事件，再定义快照和游标
4. 让前端继续基于 mock 数据开发主页面
5. 把 EventBus 直接当数据库使用

---

## 11. 推荐的交付顺序

最小可控交付建议如下：

1. 先交付“单实例 + 快照 + 命令闭环 + 事件恢复”
2. 再交付“真实 UI 接入”
3. 最后交付“多实例隔离模式”

不要把“多实例”放进第一阶段验收承诺里。

## 12. 测试矩阵

### 集成测试

### 端到端测试

1. 页面加载快照
2. WebSocket 接事件
3. 按钮发命令
4. 页面刷新恢复
5. 断线重连恢复

---

## 13. 风险管理

### 高风险项

1. 旧状态文件格式与新 schema 兼容
2. PM 主链路改动引入回归
3. WebSocket 与快照游标不一致
4. 多实例隔离工作区的合并策略

### 控制手段

1. 先收敛 schema
2. 先做单实例闭环
3. 把 mock 从主页面剥离
4. 多实例作为后续里程碑，不提前承诺

---

## 14. 结论

V2 实施计划的核心变化，是把工作顺序从“页面优先”改成“运行时一致性优先”。如果严格按本计划推进，Dashboard 将先成为可信控制台，再成为完整产品界面。

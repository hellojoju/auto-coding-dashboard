# AI全自动开发平台 - Dashboard 设计修改建议

> 评审对象: `docs/superpowers/specs/2026-04-18-dashboard-design.md`
> 评审日期: 2026-04-18
> 结论: 方案方向正确，但要先补齐状态真源、命令闭环、并发隔离、事件协议，才能把 Dashboard 做成可信的控制台，而不只是可视化界面。

## 1. 总体判断

当前 Dashboard 方案的目标定义是合理的：

1. 为多 Agent 自动开发流程提供实时可视化
2. 允许用户以甲方视角和 PM 交互
3. 支持人工审批、暂停、重试、跳过等干预动作
4. 为后续多实例并行执行提供监控和协作入口

但从当前项目实现现状看，方案存在几个关键缺口：

1. 缺少统一的状态真源
2. 控制指令没有执行闭环
3. 多实例冲突控制过于理想化
4. 实时事件协议不够稳健
5. 前后端契约和运行拓扑尚未收敛

这些问题不解决，Dashboard 很容易变成“看起来可控，实际上没有真正驱动主流程”的旁路界面。

---

## 2. 主要问题与修改建议

### 2.1 状态真源不统一

#### 问题

现有设计希望所有 Dashboard 状态落在项目 `data/` 下，并与 `features.json` 等运行数据保持一致。但当前工程里，状态读写路径和文件结构并未真正统一：

1. `ProjectManager` 将 `features.json` 写到 `project_dir/data/`
2. `FeatureTracker` 仍通过全局配置读取仓库根目录 `data/features.json`
3. `ProjectManager` 写出的 `features.json` 是裸数组
4. `FeatureTracker` 读取时要求结构为 `{ "features": [...] }`

这意味着 Dashboard 看到的状态、PM 实际调度使用的状态、测试中 patch 过的状态，可能不是同一份数据。

#### 风险

1. 看板显示与真实执行进度不一致
2. 断线重连后拿到过期快照
3. REST 操作命中错误项目或错误运行实例
4. 后续多项目支持时无法区分 project scope

#### 修改建议

1. 引入统一的 `ProjectStateRepository` 或等价存储层，明确 project-scoped 状态目录
2. 所有状态读写都通过同一个 repository 完成，不再让 `ProjectManager`、`FeatureTracker`、Dashboard 各自直写文件
3. 明确 `features.json`、`agents.json`、`chat.json`、`events.log` 的 schema，并加 `schema_version`
4. 在 Dashboard 中只消费 repository 导出的快照，不维护独立平行状态源

#### 建议落地方式

优先统一这几个对象的归属：

1. Feature 列表
2. Agent 实例状态
3. 当前运行实例标识 `run_id`
4. 聊天记录
5. 事件流

---

### 2.2 控制指令没有闭环

#### 问题

现方案定义了批准、驳回、暂停、恢复、重试、跳过、聊天等操作，但没有严格定义命令生命周期。

当前实现更接近“前端发一个 REST 请求，后端发一条事件或修改一点内存状态”，而不是“调度器接收命令并确认执行结果”。

例如：

1. `approve/reject` 只是记录一个决策事件
2. `pause/resume` 只是改 Dashboard 内存中的 agent 状态
3. `retry/skip` 只是发一个事件，但主执行循环没有消费逻辑

这会让 Dashboard 的按钮表现和真实 orchestration 行为脱节。

#### 风险

1. 用户点击“暂停”，界面显示已暂停，但任务实际仍在运行
2. 用户点击“批准”，PM 侧没有真正继续执行
3. 重试、跳过等动作无法审计，也无法确认是否执行成功
4. 操作失败时没有明确错误反馈机制

#### 修改建议

1. 把“控制动作”建模成显式命令，而不是普通事件
2. 为每个命令分配 `command_id`
3. 定义命令状态：`pending`、`accepted`、`applied`、`rejected`、`failed`
4. 由调度器或 PM 组件消费命令，再产出事实事件
5. 前端以命令结果为准更新 UI，不直接把“按钮点击”视为状态完成

#### 建议命令模型

```json
{
  "command_id": "cmd_001",
  "type": "pause_agent",
  "target_id": "backend-1",
  "issued_by": "user",
  "issued_at": "2026-04-18T14:35:00Z",
  "status": "pending"
}
```

对应事实事件示例：

```json
{
  "event_id": 128,
  "type": "agent_paused",
  "agent_id": "backend-1",
  "caused_by_command_id": "cmd_001",
  "timestamp": "2026-04-18T14:35:01Z"
}
```

### 2.3 多实例并发控制过于理想化

#### 问题

方案采用“逻辑锁 + PM 路径分配”的方式避免冲突，并明确不做物理沙箱。这在文档层面足够简单，但在 agentic coding 场景里并不稳。

主要原因：

1. Agent 不一定只改预分配路径
2. 一个 feature 很可能横跨多个目录和共享文件
3. 重构、依赖升级、代码生成会产生超出预估范围的改动
4. 当前执行模型是在共享工作区里运行，并且 git 提交粒度较粗

如果多个实例共享一个工作区，仅靠 PM 的锁表很难保证真正无冲突。

#### 风险

1. 两个实例改到同一文件但 PM 未提前识别
2. `git add -A` 将其他实例或用户的脏改动一并提交
3. `git diff HEAD` 提取的改动不属于当前 feature
4. 出现污染后很难定位是哪一个实例引入的问题

#### 修改建议

1. 逻辑锁保留，但只作为调度优化，不作为最终隔离机制
2. 多实例执行时优先使用隔离工作目录
3. 推荐方案：`git worktree`、临时 branch、或独立工作副本
4. 将产出合并回主线时再由 PM 统一做集成验证
5. 为每个实例记录独立的 workspace 标识，而不只记录逻辑路径

#### 建议原则

1. 单实例模式可以共享工作区
2. 多实例模式必须引入执行隔离
3. 只有隔离后的产物才有资格进入集成和验收阶段

---

### 2.4 事件协议缺少可恢复性和顺序保证

#### 问题

方案目前主要依赖内存队列和 `events.log`，但事件定义仍偏轻量：

1. 只有 `type` 和若干业务字段
2. 没有全局单调递增的 `event_id`
3. 重连补发依赖时间戳比较
4. EventBus 初始化时会清理日志

这会导致事件补偿和顺序恢复能力很弱。

#### 风险

1. 多事件同一时间戳时无法准确排序
2. 服务重启后无法知道前端漏了哪些事件
3. 事件日志被清空后，重连补偿失效
4. 后续若支持多项目或多运行实例，事件会混淆

#### 修改建议

1. 为每条事件增加单调递增 `event_id`
2. 增加 `project_id`、`run_id`、`schema_version`
3. 前端重连改成“拿最新快照 + 从某个 event_id 之后补增量”
4. 不要在 EventBus 初始化时主动清空日志
5. 事件日志按运行实例分段或归档，而不是简单复用一个全局文件

#### 建议事件结构

```json
{
  "event_id": 128,
  "project_id": "todo-app",
  "run_id": "run_20260418_1430",
  "type": "agent_status_changed",
  "timestamp": "2026-04-18T14:35:01Z",
  "schema_version": 1,
  "payload": {
    "agent_id": "backend-1",
    "feature_id": "F004",
    "old_status": "idle",
    "new_status": "busy"
  }
}
```

### 2.5 前后端契约和部署拓扑没有收敛

#### 问题

当前文档、计划和实现之间已经出现漂移：

1. 设计文档写的是 Next.js 前端 `:3568` + FastAPI `:8000`
2. 计划文档里的文件路径和当前仓库实际目录不完全一致
3. 当前前端目录是 `dashboard-ui`，不是计划中的 `dashboard-web`
4. 当前前端页面还在使用模拟数据启动
5. 现有 API 前缀、WebSocket 路径、代理配置并未完全收敛

这不是简单的命名问题，而是说明“谁是系统入口、谁提供状态快照、谁负责实时推送”还没有形成稳定契约。

#### 风险

1. 前端开发依赖 mock，直到联调才暴露协议问题
2. 后端改动无法及时反映到前端契约
3. 同一个项目中存在多套路径命名和运行方式，维护成本高

#### 修改建议

1. 固化一份前后端契约文档，包含：
   - REST 路由
   - WebSocket 路径
   - 事件 payload
   - 快照结构
   - 错误返回格式
2. 在代码中生成或维护共享 schema
3. 前端首屏必须先拉真实快照，再接 WebSocket
4. mock 数据只允许出现在 story/demo 环境，不进入默认主页面逻辑
5. 统一目录命名和运行入口，避免 `dashboard-web`、`dashboard-ui`、`/ws`、`/ws/dashboard` 并存

## 3. 建议的架构调整

### 3.1 调整后的职责边界

建议把架构从“Dashboard 直接挂在现有运行逻辑旁边”调整为“以调度内核为中心，Dashboard 只做可视化和命令入口”。

建议职责如下：

1. `ProjectManager`
   - 只负责调度决策
   - 消费用户命令
   - 产出调度决策和事实事件

2. `ProjectStateRepository`
   - 持久化 features、agents、commands、events、chat
   - 对外提供快照读取接口

3. `EventBus`
   - 只负责内存广播
   - 不承担业务状态真源职责

4. `Dashboard API`
   - 读取 repository 快照
   - 接收用户命令
   - 订阅事实事件并推送前端

5. `AgentPool`
   - 管理实例生命周期和隔离工作区
   - 不直接承担业务审计职责

### 3.2 推荐数据分层

建议把数据拆成三类：

1. Snapshot 数据
   - 当前项目状态
   - 当前 agents 状态
   - 当前 features 状态

2. Event 数据
   - 已发生的事实
   - 只追加，不覆盖

3. Command 数据
   - 用户或系统发出的控制意图
   - 有单独状态机

这样 Dashboard 的“显示”和“控制”才会清晰分离。

## 4. 实施优先级建议

### P0：必须先改

1. 统一状态真源和项目目录作用域
2. 统一 `features.json` 等核心文件 schema
3. 明确命令模型，并让 PM/调度器真正消费
4. 为事件引入 `event_id`、`run_id`

### P1：建议在多实例前完成

1. 引入实例级隔离工作区
2. 调整 git 提交策略，避免共享工作区下 `git add -A`
3. 收敛 REST/WebSocket 契约
4. 用真实状态替换主页面 mock 数据

### P2：体验和可维护性优化

1. 增加操作审计视图
2. 增加命令执行状态反馈
3. 增加按 `run_id` 筛选历史记录
4. 增加前端共享类型定义或 schema 校验

---

## 5. 建议的方案修订方向

如果要继续保留当前设计的大方向，建议把原设计文档修订成下面的思路：

1. Dashboard 是调度内核的可视化层，不是独立状态中心
2. EventBus 负责广播，不负责定义状态真相
3. 命令与事件分离，按钮操作必须可追踪、可确认、可失败
4. 单实例与多实例采用不同执行隔离策略
5. 前端永远基于快照初始化，再通过增量事件更新

---

## 6. 最终结论

这份设计方案的产品方向没有问题，甚至很有必要。但如果按当前版本直接推进实现，最容易先做出来的是“有界面的 Dashboard”，而不是“可信的项目控制台”。

建议先修订基础架构假设，再继续推进 UI 与交互层。更具体地说，应优先解决：

1. 状态真源
2. 命令闭环
3. 并发隔离
4. 事件可恢复性

在这四件事收敛之前，不建议把主要精力投入到按钮细节和界面 polish。

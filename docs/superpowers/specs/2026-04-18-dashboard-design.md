> 日期: 2026-04-18
> 状态: 设计完成，待实现

---

## 1. 需求概述

### 1.1 背景

当前系统已有完整的 Agent 编排骨架（PM + 9 个子 Agent），但缺乏可视化监控手段。用户只能通过终端日志了解进展，无法实时看到 Agent 状态、无法干预执行方向、无法像甲方 review 一样跟 Team Leader（PM）对话。

### 1.2 核心需求

1. **实时状态监控**：每个 Agent 的运行状态（运行中/等待权限/等待PM指令/完成/阻塞/暂停）
2. **甲方交互**：跟 PM 对话，了解进展，审查产出，批准/驳回/修改下一步计划
3. **操作控制**：常用操作做成按钮（批准/驳回/暂停/重试/跳过），复杂需求通过对话输入
4. **多实例支持**：同一角色可有多个 Agent 实例（如 2 个前端、2 个后端）并行工作
5. **模块隔离**：多实例之间的文件修改不冲突，PM 负责任务划分和接口协调

### 3.2 操作按钮清单

| 位置 | 按钮 | 作用 |
|------|------|------|
| 进行中卡片 | 日志 | 侧边弹出该 Agent 的实时日志面板 |
| 进行中卡片 | 暂停 | 暂停当前 Agent 执行 |
| 已完成卡片 | 查看产出 | 打开该 Feature 产出的代码文件列表 |
| 阻塞卡片 | 重试 | 重新执行失败任务 |
| 阻塞卡片 | 跳过 | 跳过该 Feature 继续其他任务 |
| PM 对话区 | 批准 | 确认 PM 的下一步计划 |
| PM 对话区 | 驳回 | 要求 PM 重新规划 |
| PM 对话区 | 修改方向 | 打开输入框，输入新的任务方向 |

### 3.3 对话区交互

- **PM 主动推送**：当 PM 做出调度决策（如"准备启动 F007"）时，推送到对话区并等待批准
- **用户主动发起**：在输入框输入任何指令，如 "F007 先不做支付，改成做积分系统"
- **快捷按钮**：批准/驳回/修改方向始终显示在最新 PM 消息下方
- **历史消息**：可滚动查看之前的对话记录

## 4. 多实例 Agent 协调

### 4.1 核心问题

同一角色的多个 Agent 实例（如 Frontend#1 和 Frontend#2）同时工作时，如果修改同一个文件会产生冲突。

### 4.2 解决方案：逻辑锁 + PM 路径分配

**不搞物理沙箱**（Agent 直接写到正确的目标路径），而是通过 PM 调度层面保证不冲突：

1. PM 在分配 Feature 时，明确指定该 Feature 涉及的文件路径
2. PM 维护一个"文件锁表"：`{path: agent_id}`，记录哪些文件正在被哪个 Agent 修改
3. 如果两个 Feature 涉及同一文件，PM 确保它们串行执行（有隐式依赖）
4. 如果两个 Feature 涉及不同文件，PM 可以并行分配给不同实例

### 4.3 实例池管理

```
AgentPool:
  backend: [Backend#1 (idle), Backend#2 (busy-F008)]
  frontend: [Frontend#1 (busy-F005), Frontend#2 (idle)]
  database: [Database#1 (idle)]
  ...
```

**实例数量规则**：
- 默认：每角色 1 个实例
- PM 自动分析：如果某角色有 N 个无依赖的 ready Feature，自动申请第 2 个实例
- 上限：backend 最多 3 个，frontend 最多 3 个，其他角色始终 1 个

### 4.4 PM 分配逻辑

```
1. 获取下一个 ready 的 Feature
2. 查找该角色对应的 Agent Pool 中的 idle 实例
3. 检查该 Feature 涉及的文件是否与正在执行的 Feature 冲突
4. 如果不冲突 → 分配给 idle 实例
5. 如果冲突 → 标记为 blocked，等前一个完成
6. 如果所有实例都 busy → 等待
```

## 5. 实时通信协议

### 5.1 WebSocket 事件（后端 → 前端推送）

| 事件类型 | 数据字段 | 触发时机 |
|---------|---------|---------|
| `agent_status_changed` | `agent_id, feature_id, old_status, new_status` | Agent 状态变化 |
| `agent_log` | `agent_id, feature_id, message, timestamp` | Agent 执行中产生日志 |
| `feature_completed` | `feature_id, files_changed, summary` | Feature 完成 |
| `pm_decision` | `decision, next_actions, requires_approval` | PM 做出调度决策 |
| `error_occurred` | `feature_id, agent_id, error_message` | 执行出错 |

### 5.2 REST API 端点（前端 → 后端操作）

| 端点 | 方法 | 作用 |
|------|------|------|
| `/api/approve` | POST | 批准 PM 的下一步计划 |
| `/api/reject` | POST | 驳回，要求重新规划 |
| `/api/chat` | POST | 给 PM 发消息对话 |
| `/api/pause` | POST | 暂停指定 Agent |
| `/api/resume` | POST | 恢复指定 Agent |
| `/api/retry` | POST | 重试失败 Feature |
| `/api/skip` | POST | 跳过阻塞 Feature |
| `/api/state` | GET | 获取当前完整状态快照（断线重连用） |

### 5.3 心跳与重连

- 前端每 30 秒发送 ping，后端回复 pong
- WebSocket 断线后自动重连
- 重连成功后，后端推送完整状态快照（通过 `/api/state` 数据），避免漏掉事件

## 6. 数据模型

所有数据存储在项目的 `data/` 目录下，与现有 `features.json` 保持一致。

### 6.1 Agent Instance (`data/agents.json`)

```json
{
  "id": "backend-1",
  "role": "backend",
  "instance_number": 1,
  "status": "idle|busy|paused|error",
  "current_feature": "F004",
  "total_tasks_completed": 5,
  "started_at": "2026-04-18T10:00:00Z"
}
```

### 6.2 Feature Task（扩展现有 `data/features.json`）

新增字段：
```json
{
  "assigned_instance": "backend-1",
  "workspace_path": "src/api/",
  "files_changed": ["src/api/users.py"],
  "started_at": "2026-04-18T14:20:00Z",
  "completed_at": "2026-04-18T14:32:00Z"
}
```

### 6.3 Event Log (`data/events.log`)

追加写入，每行一个 JSON 对象：
```json
{"type":"agent_log","agent_id":"backend-1","feature_id":"F004","message":"创建了 GET /users 端点","timestamp":"2026-04-18T14:32:01Z"}
```

- 用于实时推送（内存队列）+ 持久化（文件追加）
- 断线重连时从文件读取缺失的事件
- 项目初始化时清空

### 6.4 Chat History (`data/chat.json`)

```json
{
  "id": "chat_001",
  "role": "user|pm",
  "content": "F007 先不做支付，改成做积分系统",
  "timestamp": "2026-04-18T14:35:00Z",
  "action_triggered": "override"
}
```

## 7. 技术栈

| 层 | 技术 | 理由 |
|----|------|------|
| 前端框架 | Next.js 14 (App Router) | 成熟生态，SSR 可选，与现有 Python 后端解耦 |
| UI 组件 | shadcn/ui + Tailwind CSS | 快速搭建美观看板，可定制性强 |
| 实时通信 | WebSocket (ws 库) | 双向实时，低延迟 |
| 状态管理 | Zustand | 轻量，适合实时数据流 |
| 后端框架 | FastAPI（已有） | 项目本身是 Python/FastAPI，天然契合 |
| 事件总线 | 内存队列 + 文件追加 | 简单够用，不需要引入 Redis 等外部依赖 |
| 开发服务器 | Next.js dev server on :3568 | 用户指定端口 |

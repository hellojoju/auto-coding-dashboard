# auto-coding 平台 V2 设计规格

> 日期：2026-04-19
> 状态：待用户审查后进入实现阶段

## 1. 项目概述

构建一个 PM 为中心的多 Agent 软件开发平台，用户（甲方）只需一句话需求，PM 自动拆解任务、调度子 Agent、审查交付物、迭代修复，全程可视化监控。

## 2. 核心架构决策

### 2.1 技术栈
- **执行引擎**: Claude Code 子进程（`claude --print -p <prompt>`）
- **上下文机制**: 基于文件系统状态，非长对话记忆。每次调用读取文件内容 + git diff 作为上下文
- **后端**: FastAPI + WebSocket 实时推送
- **前端**: Next.js 16.2.4 + Zustand + Tailwind CSS v4
- **通信**: EventBus 内部广播 + REST API + WebSocket 推送到前端

### 2.2 PM 是信息枢纽
- 所有信息流向 PM，由 PM 决定传递给谁（一对一或一对多）
- 取消 Agent 间直接通信，避免信息爆炸
- PM 是唯一 Team Leader，用户只跟 PM 交互

### 2.3 工作包（Work Package）调度
- PM 定义 work package：依赖关系、交付物、验收标准
- 按依赖顺序派发，前置未完成则等待
- 每个 work package 是独立的 `claude -p` 调用

### 2.4 Step 粒度控制
- PM 控制执行粒度，每步是独立调用
- 每步有明确输入/输出/验收标准
- 支持单步重试、跳过、回滚

## 3. Agent 角色体系

### 3.1 PM（项目经理兼需求分析师）
- **需求分析阶段**: 接收用户输入，理解意图，拆解需求
- **任务调度阶段**: 定义 work package，派发给子 Agent
- **审查阶段**: 五阶段审查（测试 → 架构 → 功能 → 安全 → UI）
- **决策权**: 对 Agent 的优化建议有取舍权，不盲从
- **原则**: 有自己的思路，不顺着其他 Agent 的思路走

### 3.2 子 Agent 列表
| 角色 | 文件 | 多实例 |
|------|------|--------|
| 架构师 | `agents/architect.py` | 否 |
| 后端开发 | `agents/backend_dev.py` | 是 (2-3) |
| 前端开发 | `agents/frontend_dev.py` | 是 (2-3) |
| 数据库专家 | `agents/database_expert.py` | 否 |
| QA 测试 | `agents/qa_tester.py` | 否 |
| 安全审查 | `agents/security_reviewer.py` | 否 |
| UI 设计师 | `agents/ui_designer.py` | 否 |
| 文档工程师 | `agents/docs_writer.py` | 否 |
| 技术审查 | 独立 Agent | 否 |

### 3.3 技术审查独立
- 代码质量审查从 PM 剥离
- PM 做决策，技术审查提供专业意见
- 避免 PM 提示词过载

## 4. PM 灵魂移植

从历史对话中提取用户的：
- 决策模式（什么时候拍板，什么时候追问）
- 审查优先级（最关心什么，最不能容忍什么）
- 沟通风格（简洁直接，不要啰嗦）
- 工作习惯（先看结果再问细节，不要主动汇报过程）

写入 `prompts/product_manager.md`。

## 5. 五阶段审查流水线

按优先级顺序：
1. **测试通过** — 单元测试跑通，覆盖率达标
2. **架构合理** — 分层清晰，无循环依赖，接口设计合理
3. **功能完整** — 需求覆盖，无遗漏
4. **安全合规** — 无注入、无硬编码密钥、权限正确
5. **UI 达标** — 视觉一致，交互流畅

任一阶段不通过 → 打回对应 Agent → 修复 → 重新审查

## 6. 前端看板设计

### 6.1 布局
```
┌─────────────────────────────────────────────────────┐
│ 顶部: 项目卡片 + 连接状态 + 进度 + 执行控制          │
├──────────────────────┬──────────────────────────────┤
│ 左侧: 实时活动流      │ 右侧: 看板 (Kanban)           │
│ (Agent 动作实时显示)  │ (待排→进行→审查→完成→阻塞)    │
├──────────────────────┤                              │
│ 底部左: Agent 状态面板│                              │
│ (运行中/空闲/等待/错误)│                              │
├──────────────────────┴──────────────────────────────┤
│ 可折叠面板: 一句话需求 + 对话窗口                     │
│ (默认收起，展开后显示需求输入+对话历史+发送框)         │
└─────────────────────────────────────────────────────┘
```

### 6.2 设计原则
- **监控优先**: 80% 区域用于监控，20% 用于交互
- **实时活动流**: 类似终端输出但结构化，用户第一眼关注
- **可折叠对话**: 默认收起，需要时才展开
- **状态实时同步**: WebSocket 推送 Agent 状态变更

### 6.3 Agent 状态
- 🟢 **运行中**: 正在执行任务
- 🟡 **等待指令**: 完成任务，等 PM 派发新任务
- 🟠 **等待审查**: 提交交付物，等 PM 审查
- 🔴 **错误**: 执行失败，需要重试
- ⚪ **空闲**: 无任务分配

## 7. 执行控制

通过 `PMCoordinator` 的线程安全方法实现：
- `start_execution()` — 启动后台协调循环
- `stop_execution()` — 请求停止
- `get_execution_status()` — 查询当前状态

REST API 端点：
- `POST /api/execution/start`
- `POST /api/execution/stop`
- `GET /api/execution/status`

前端通过 Zustand store 同步状态，页面头部显示启停按钮。

## 8. 上下文验证结论

实际测试确认：
- `claude -p` 读取文件系统状态（文件内容、git diff）作为上下文
- 对话历史是部分的，不完整的
- 最近的操作会被记住，早期的会被遗忘
- **结论**: 上下文来自文件，不来自对话记忆。所有项目状态必须落盘
